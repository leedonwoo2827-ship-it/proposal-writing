# AI 제안서 작성 스킬 (Skill 3)

## 개요

이 스킬은 **Skill 1(RFP 뼈대)** + **Skill 2(회사 데이터)**를 결합하여 제안서 초안을 작성하고,
**마크다운 형식(proposal-body.md)**으로 출력합니다.

---

## ⚠️ 핵심 원칙: 출력 형식

> **Skill 3의 최종 결과물은 반드시 `proposal-body.md` 파일로 저장되어야 한다.**

이 마크다운 파일은 hwpx-writer MCP의 `md_to_hwpx` 도구로 전달되어 HWPX 문서로 변환된다.

---

## 주요 기능

### 1. 데이터 기반 컨텐츠 합성

**입력:**
- **Skill 1**: RFP에서 추출한 Backbone (섹션 구조, depth 1~2)
- **Skill 2**: Reference 제안서 + Raw 데이터 (수치, 팩트)

**처리:**
- Reference의 논리 구조와 톤 학습
- Raw 데이터의 핵심 키워드/수치 추출
- 섹션별로 적절한 데이터 매칭

**출력:**
- 마크다운 파일 (`proposal-body.md`)

---

### 2. 분량 최적화 엔진

설정된 프리셋(5만~20만 자)에 맞춰:
- 중요 섹션 → Reference의 상세한 논리 전개
- 일반 섹션 → 요약된 사실 중심 서술

---

### 3. 근거 기반 컬러링 (인라인 마커)

출처 텍스트를 **문장 안에 자연스럽게 녹여** 넣고, 구절 단위로 색상 마커를 표시한다:
- 🔴 `{{red:구절}}`: Raw 데이터 기반 텍스트
- 🟢 `{{green:구절}}`: Reference/RFP 기반 텍스트
- 마커 없음: AI 생성 텍스트 (기본 검정)

마크다운 예시:
```markdown
본 훈련과정은 {{red:20010105 빅데이터분석 세분류}}를 적용하며, {{green:산업 현장의 실무 역량 강화}}를 핵심 목표로 설계되었다.
```

> **핵심 원칙 — 융합 작성**: 출처 텍스트를 별도 블록으로 나열하지 않는다.
> 반드시 AI가 작성한 문장 흐름 안에 출처 구절을 자연스럽게 섞어 넣는다.

## 마크다운 작성 규칙

### 헤딩 구조 (depth vs 스타일 레벨)

> **용어 구분**
> - **depth**: RFP 뼈대의 계층 구조 (Lv1 섹션, Lv2 하위섹션). rfp-analysis에서 추출한 구조.
> - **스타일 레벨**: hwpx-writer가 렌더링에 사용하는 폰트/기호 설정 (level 1~6).

마크다운 헤딩과 HWPX 스타일 레벨의 대응:

| 마크다운 작성 방법 | HWPX 렌더링 | 스타일 레벨 | 폰트 |
|---|---|---|---|
| `# 문서 제목` (첫 줄 1회만) | 문서 제목 | title | Noto Sans KR Bold, 22pt(12pt) 중앙 |
| `# 1. 섹션명` 또는 `## 1. 섹션명` | 섹션 제목 | level 1 | Noto Sans KR Bold, 16pt(12pt) |
| `### 1.1 소항목명` | 소항목 제목 | section_subtitle | Noto Sans KR Bold, 15pt(11pt) |
| `#### 항목명` | 항목명 | level 4 | Noto Serif KR, 10pt |
| 일반 단락 텍스트 (헤딩 없음) | (기호 없음) 단락 | level 2 | Noto Serif KR, 10pt |
| `◻ 본문 항목` (직접 기호 사용) | ◻ 항목 | level 3 | Noto Serif KR, 10pt (내어쓰기 15pt) |
| `○ 세부 항목` (직접 기호 사용) | ○ 항목 | level 4 | Noto Serif KR, 10pt |
| `― 세부 설명` (직접 기호 사용) | ― 세부 설명 | level 5 | Noto Serif KR, 10pt |
| `※ 참고` (직접 기호 사용) | ※ 참고 | level 6 | Noto Serif KR, 10pt |

> **⚠️ 중요: 기호 직접 사용 원칙**
>
> **기호 체계를 proposal-styles.json과 동일하게 사용하세요:**
> - Level 3 (본문 항목): `◻` (White Medium Square, U+25FB) — 내어쓰기 15pt 적용
> - Level 4 (세부 항목): `○`
> - Level 5 (보충 설명): `―`
> - Level 6 (참고/주석): `※`
>
> **작성 방법:**
> - 마크다운 `-` 리스트 문법 대신 **기호를 직접 텍스트 앞에 붙여** 작성합니다
> - 예: `◻ COURSEMOS 플랫폼 구축: 오픈소스 LMS 기반...`
> - hwpx-writer는 기호를 자동 감지하여 올바른 스타일 레벨을 적용합니다
>
> **절대 사용 금지:**
> - `□` (Black Square, U+25A1) — 한글 오피스에서 내어쓰기가 무시됨
> - `●` (검은 원) — 사용하지 않음
> - `-` 마크다운 리스트 문법 — 표 안에서 혼란 초래 가능

### 볼드(`**`) 처리

> ⚠️ **`**볼드**` 마크다운은 사용하지 않는다.**
> 강조가 필요하면 색상 마커(`{{red:}}`, `{{green:}}`) 또는 헤딩 레벨을 활용한다.
> 표 안에서도 `**`는 절대 사용하지 않는다.

### 작성 예시

```markdown
# 파이썬과 머신러닝으로 완성하는 경영데이터 분석

# 1. 훈련과정 개요

### 1.1 훈련과정명

#### 과정 명칭의 정의 및 기획 배경

본 훈련과정은 {{red:AI 기초(10차시), AI 심화(10차시)}} 등 총 5개 과정을 통합·재구성하였습니다.
특히 {{green:SRT 기반 자막 데이터로 전 차시가 체계적으로 구조화}}되어 있습니다.

◻ 데이터 수집 및 정제: SQL 활용 데이터 추출, 데이터 정제 및 전처리
◻ 통계 분석 및 시각화: 통계적 검증, 인사이트 도출

― 직무 역량 매핑 체계

본 과정의 학습 내용은 산업 현장의 구체적 직무와 다음과 같이 매핑됩니다.
```

## 스타일 레벨 참조표

> level 1만 제목체(돋움체). level 2~6은 본문(바탕체).

| 스타일 레벨 | 기호 (직접 사용) | 폰트 | 용도 |
|------|------|------|------|
| level 1 | (없음) | Noto Sans KR Bold, 16pt(12pt) | 섹션/소항목 제목 |
| level 2 | (없음) | Noto Serif KR, 10pt | 일반 본문 (leftMargin 0) |
| level 3 | ◻ (White Square U+25FB) | Noto Serif KR, 10pt | 본문 항목 (내어쓰기 15pt, leftMargin 0) |
| level 4 | ○ | Noto Serif KR, 10pt | 세부 항목 (내어쓰기 15.3pt, leftMargin 4) |
| level 5 | ― | Noto Serif KR, 10pt | 보충 설명 (내어쓰기 15.3pt, leftMargin 8) |
| level 6 | ※ | Noto Serif KR, 10pt | 참고/주석 (내어쓰기 15.3pt, leftMargin 12) |
| section_subtitle | (없음) | Noto Sans KR Bold, 11pt | 소항목 제목 (### 사용) |

## 색상 마커 (인라인 방식)

출처 구별은 `text` 필드 안에 인라인 마커로 표현한다:

| 출처 | 마커 형식 | 설명 |
|------|-----------|------|
| Raw 데이터 (수치, 팩트) | `{{red:구절}}` | 🔴 적색 |
| Reference / RFP 기반 | `{{green:구절}}` | 🟢 녹색 |
| AI 생성 텍스트 | 마커 없음 | ⚫ 기본 검정 |

hwpx-writer가 마커를 인식하여 해당 색상의 글자로 자동 변환한다.

### 융합 작성 규칙
1. 문장 단위로 섞어 쓴다 — AI 텍스트(검정)와 출처 텍스트(적색/녹색)가 하나의 문장 안에서 이어져야 한다.
2. 출처 텍스트를 문장의 일부로 편입한다 — 앞뒤에 접속사, 서술어, 맥락 설명을 붙여 완성된 문장으로 만든다.
3. 절대로 출처 텍스트만 따로 블록으로 나열하지 않는다.
4. 마커는 구(phrase) 또는 절(clause) 단위로 감싼다 — 문장 전체가 아니라 핵심 구절만 마킹한다.

## ⚠️ 표(Table) 작성 필수 규칙

> **표는 HWPX/PDF 변환 시 가장 민감한 요소이다. 아래 규칙을 반드시 준수한다.**

### 셀 내용 규칙
1. **셀 하나에 색상 마커는 하나만** — `{{red:텍스트}}` 하나. 중첩이나 복수 마커 금지.
2. **셀 전체를 마커로 감쌀 수 있다** — 셀 내용 전체가 같은 색이면 셀 전체를 감싼다.
3. **마커 안에 `|` 파이프 문자 금지** — 표 파싱이 깨진다.
4. **빈 셀은 `-` 로 표시** — 완전히 비운 `| |`은 사용하지 않는다.
5. **`**볼드**` 마크다운은 표 안에서 절대 사용 금지** — 볼드 불필요.

### 표 구조 규칙
1. **헤더 행 필수** — `| 구분 | 내용 |` + `|---|---|` 형식.
2. **구분선은 `|---|` 형식** — 콜론(`:`) 정렬은 무시됨.
3. **컬럼 수 일치** — 모든 행의 `|` 개수가 헤더와 동일해야 한다.
4. **표 위 짧은 텍스트는 캡션으로 자동 승격** — 표 바로 위에 60자 미만 텍스트가 있으면 표 제목이 된다.

### 표 안 색상 마커 예시

올바른 예:
```markdown
| 구분 | 내용 | 비고 |
|---|---|---|
| 장기간 프로젝트 | {{red:48개월 장기 프로젝트 수행에 따른 위험 요소}} | 높음 |
| 핵심 추진 방향 | {{green:프로젝트 관리 전문팀 구성}} | 필수 |
```

잘못된 예 (금지):
```markdown
| {{red:{{green:내용}}}} | 텍스트 |          ← 마커 중첩 금지
| **구분** | {{red:내용}} |                  ← 볼드 사용 금지
| 항목 |  |                                  ← 빈 셀은 - 로 채울 것
| {{red:텍스트1}} {{green:텍스트2}} | 비고 | ← 한 셀에 복수 마커 금지
```

## 품질 보증

### 검증 항목

- ✅ 모든 섹션에 최소 1개 이상의 본문 항목 존재
- ✅ `{{red:}}`/`{{green:}}` 마커가 구절 단위로 적용
- ✅ 전체 글자 수가 목표 범위(±10%) 내
- ✅ `**볼드**` 미사용 (색상 마커 또는 헤딩으로 대체)
- ✅ 표 셀에 마커 중첩 없음 (1셀 1색상 마커)
- ✅ 표 안 `**볼드**` 절대 미사용
- ✅ 표 구조 정합성 (컬럼 수 일치, 빈 셀 `-` 처리)
